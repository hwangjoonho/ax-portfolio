Problem Solving
핵심 문제 해결 과정
1

base64 인코딩 — 33% 토큰 낭비

Problem

MCP 프로토콜은 JSON-RPC 기반으로 텍스트만 전송 가능. 이미지를 보내려면 base64 인코딩이 필수.

1MB 이미지 → 1.33MB 문자열 (+33%). 5MB 이상은 타임아웃/메모리 오류. 인코딩→전송→디코딩→재전송으로 이중 변환 발생.

Solution

공유 tmpfs 볼륨 /shared 도입. 3개 컨테이너(MCP, Proxy, vLLM)가 RAM 디스크를 공유.

MCP가 이미지를 /shared에 저장 → Proxy가 같은 파일을 전처리 → vLLM이 file:// URL로 직접 읽기.

네트워크 상 이미지 바이너리 전송 = 0회, base64 변환 = 0회
2

Claude Desktop — multipart 전송 불가

Problem

Claude Desktop은 MCP 경유만 가능 → JSON 텍스트 전용 → base64 필수 (문제 1로 회귀).

Trial & Error

시도: Custom Skill로 Code Execution VM에서 curl http://... 직접 전송

실패: VM은 HTTPS만 허용, HTTP 차단됨

Solution

K8s rag-mcp 서버에 HTTPS 프록시 엔드포인트 POST /upload 추가 (v4.3). multer로 multipart 수신 → 기존 함수 재사용하여 DGX3로 중계.

Claude Desktop VM → HTTPS multipart → rag-mcp → DGX3. base64 변환 0회.
3

Skill 패키징 — 4번의 시행착오

Iteration

공식 package_skill.py로 패키징, HTTPS 엔드포인트로 전송

4

VLM 할루시네이션 — 이상 문자 출력

Problem

한국어 문서 OCR 시 중국어/아랍어 아티팩트 간헐 출력. 다국어 학습 데이터에 의한 할루시네이션.

예: "합계 금액 ¥12,500 المجموع"

Solution: 3-Stage Quality Gate
Stage 1 · Prevention
프롬프트 방어
"Output only recognized text without location coordinates or special tokens"
Stage 2 · Cleanup
clean_ocr_output()
<|LOC_xxx|> 토큰 제거, 다중 공백/개행 정리
Stage 3 · Auto Retry
detect_anomaly_scripts()
문자별 스크립트 분류 → 이상 비율 >3% 시 language 힌트 포함 2차 OCR
5

소형 이미지 정확도 저하

Problem

654×445 이미지 → VLM 비전 인코더 해상도 부족 → 오인식 다수

Solution: Sidecar Preprocessing Pipeline
1

FormatConverter

WEBP / HEIC / BMP → RGB 변환

2

ResolutionNormalizer

장변 <1500px → 2× LANCZOS 업스케일 (654×445 → 1308×890)

3

OrientationCorrector

PP-LCNet ONNX (6.78MB) · confidence ≥0.8 시 자동 회전

4

PerspectiveDewarper (비활성)

UVDoc ONNX · 왜곡 보정 · env로 ON 가능

설계 원칙: 이진화/그레이스케일/노이즈 제거는 하지 않음 (VLM에겐 역효과). 해상도 관리가 핵심. 각 단계 실패 시 원본 반환.

Key Insight
MCP 프로토콜 한계 우회 전략

MCP Protocol Limitation

JSON-RPC = 텍스트 전용. 바이너리 전송 불가. (SEP-1306 표준화 진행 중, 미확정)
